# 演示方案与测试用例（基于当前项目状态）

## 1. 是否必须全部任务训练完成才能演示？
不需要。

根据 README 设计：每个任务是独立的 `train -> inference -> app`，可以单独演示。只要某个任务的模型权重存在，就能先演示该任务。

## 2. 演示是否需要展示 loss？
结论：建议展示，但不要作为主演示内容。

推荐做法（两层展示）：
1. 第一层（面向效果）：只展示输入/输出可视化结果，保证“能跑、能看效果”。
2. 第二层（面向可信度）：在页面或演示记录中补充训练日志摘要（如 `train_loss / eval_loss / epoch`），用于说明模型确实训练过。

你当前目标是“保证运行并能看到效果”，第一层可先完成；loss 可作为附加信息放在文档里，不阻塞演示。

## 3. 同时后台运行 10 个 Gradio + 推理会不会卡？
会，且通常会明显卡顿。

原因：
- 每个任务进程都占用内存；部分任务会占用 GPU 显存。
- 多个 Gradio 服务并发会增加 CPU 调度、I/O 与网络开销。
- 任务 2/3/8/9 等图像/语音任务推理相对重，叠加后更容易抖动。

建议方案：
1. 不要同时开 10 个 UI；采用“单任务演示轮转”。
2. 最多并行 2 个 UI（1 个当前演示 + 1 个备用）。
3. 每次演示结束后关闭当前 UI，再启动下一个任务 UI。
4. 如果必须并行，优先把重任务（2/3/8/9）与轻任务（4/5/6/10）错开。

## 5. 每个任务的演示测试用例（只要求可运行、有效果）

### Task 01 图像分类
- 启动：`python start_ui.py --task 1`
- 用例：上传一张常见物体图片（如猫/狗/汽车）。
- 预期：返回类别预测标签与概率（或分数），页面无报错。

### Task 02 目标检测
- 启动：`python start_ui.py --task 2`
- 用例：上传包含 1-3 个人体目标的街拍图。
- 预期：输出图中有检测框（bbox）与类别，至少能框出主要目标。

### Task 03 语义分割
- 启动：`python start_ui.py --task 3`
- 用例：上传含前景目标（宠物/人像）的图片。
- 预期：输出分割掩码或叠加图，前景区域有明显分割结果。

### Task 04 情感分析（当前未检测到训练权重）
- 启动：`python start_ui.py --task 4`
- 用例：
  - 正向文本：`This movie is wonderful and touching.`
  - 负向文本：`This is boring and a waste of time.`
- 预期：分别输出正向/负向倾向。
- 备注：若提示模型不存在，需先补训 task4。

### Task 05 机器翻译
- 启动：`python start_ui.py --task 5`
- 用例：输入 `I love deep learning and computer vision.`
- 预期：输出法语翻译文本，语义基本正确。

### Task 06 命名实体识别
- 启动：`python start_ui.py --task 6`
- 用例：`Barack Obama visited Paris in 2015.`
- 预期：识别出人名、地点、时间等实体标签。

### Task 07 文本摘要
- 启动：`python start_ui.py --task 7`
- 用例：输入一段 8-12 句英文新闻文本。
- 预期：输出更短摘要，保留主要信息点。

### Task 08 语音识别（当前未完成）
- 启动：`python start_ui.py --task 8`
- 用例：上传 1 秒英文命令词音频（如 `yes/no/up/down`）。
- 预期：输出命令类别。
- 备注：当前任务8未完成训练，若失败属预期，待训练完成后复测。

### Task 09 图像生成
- 启动：`python start_ui.py --task 9`
- 用例：点击生成（或输入随机种子后生成）。
- 预期：生成手写数字风格图像（MNIST 风格），可见数字轮廓。

### Task 10 时间序列预测
- 启动：`python start_ui.py --task 10`
- 用例：使用默认历史序列，预测未来若干步。
- 预期：输出预测曲线，趋势与历史序列连续、平滑。

## 6. 推荐演示顺序（低风险）
`10 -> 1 -> 5 -> 6 -> 7 -> 9 -> 2 -> 3 -> 4 -> 8`

理由：先轻量稳定任务，再展示重任务；把当前存在不确定性的 `4/8` 放后面。

## 7. 最小执行策略（保证体验）
1. 只开一个 Gradio UI。
2. 每次切任务前先关闭上一个 UI。
3. 演示时每个任务只跑 1-2 个测试用例，不做压力测试。
4. 记录失败任务与报错，演示后再补训或修复。
